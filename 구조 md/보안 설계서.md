# 보안 설계서

## 1. 목적
- 인증/인가, 데이터 보호, 서비스 간 통신 보안을 표준화한다.
- SaaS 운영에서 필요한 최소 보안 통제(접근제어, 암호화, 감사)를 확보한다.

## 2. 보안 범위
- 애플리케이션: `auth-service`, `ledger-service`, `ocr-ai-service`
- 인프라: VPC, ECS, RDS, S3, ALB, CloudFront
- 외부연동: Stripe Webhook

## 3. 위협 모델(요약)
- 계정 탈취: 약한 JWT 관리, 비밀번호 정책 미흡
- API 오남용: 인증 없는 엔드포인트 남용, 과도한 요청
- 데이터 유출: 평문 시크릿/버킷 퍼블릭/과도한 IAM 권한
- 결제 위변조: Stripe webhook 서명 검증 실패
- 내부 확산: 서비스 간 과도한 네트워크 허용

## 4. 인증/인가 설계
- 인증
  - JWT Bearer 사용
  - 만료시간 설정(현재 60분)
  - refresh token 도입 권장
- 인가
  - 사용자 리소스 접근 시 `user_id` 스코프 강제
  - PRO 기능은 `plan` 기반 권한 체크 유지
- 비밀번호
  - bcrypt 해시 저장
  - 최소 길이/복잡도 정책 강화 권장

## 5. 네트워크 보안
- VPC 계층 분리
  - Public: ALB/NAT
  - Private-App: ECS
  - Private-Data: RDS
- Security Group
  - ALB -> ECS 특정 포트만 허용
  - ECS -> RDS 3306만 허용
  - 외부에서 RDS 직접 접근 차단
- 내부 서비스 통신
  - 가능하면 Private DNS + SG 기준 최소 허용

## 6. 데이터 보호
- 전송구간 암호화
  - 외부 트래픽 HTTPS(TLS 1.2+)
  - 내부 통신도 TLS 적용 검토
- 저장구간 암호화
  - RDS 암호화(KMS)
  - S3 SSE-KMS
  - EBS 암호화
- 백업
  - RDS 자동백업 + 스냅샷 보관
  - 백업 접근권한 분리

## 7. 시크릿 관리
- 저장 위치
  - AWS Secrets Manager/SSM Parameter Store
- 접근 통제
  - ECS Task Role 최소 권한
  - 서비스별 필요한 시크릿만 부여
- 운영 원칙
  - 키/토큰 정기 교체
  - 키 노출 탐지(GitHub secret scanning)

## 8. API 보안
- 입력 검증
  - Pydantic 스키마 강제
  - 파일 업로드 크기/확장자 제한
- Rate Limit
  - ALB/WAF 또는 API Gateway 계층에서 요청 제한
- 에러 처리
  - 내부 예외/스택트레이스 외부 노출 금지
- CORS
  - 운영 도메인만 allowlist

## 9. Stripe Webhook 보안
- `stripe-signature` 검증 필수
- endpoint secret은 Secrets Manager 보관
- 재전송 이벤트(idempotency) 처리 권장
- 허용 이벤트 타입 allowlist 관리

## 10. IAM 정책 원칙
- Least Privilege
- 서비스 계정 분리
  - `task-role-auth`
  - `task-role-ledger`
  - `task-role-ocr`
- 금지
  - `*` 리소스 전체 허용
  - 관리자 권한 장기 사용

## 11. 로깅/감사
- 감사 로그
  - 로그인, 비밀번호 변경, 계정 삭제, 결제 상태 변경
- 인프라 감사
  - CloudTrail 활성화
- 보안 경보
  - GuardDuty/Security Hub 연동 권장

## 12. 취약점 관리
- 이미지 스캔
  - ECR 이미지 스캔 활성화
- 의존성 스캔
  - pip-audit/safety 또는 Snyk
- 정기 점검
  - 월 1회 취약점 패치 윈도우

## 13. 사고 대응(Incident Response)
1. 탐지: 알람/로그 기반 이상행위 식별
2. 격리: 영향 서비스/키 즉시 차단
3. 복구: 안전한 버전 재배포, 키 교체
4. 보고: 원인/영향/재발방지 문서화
