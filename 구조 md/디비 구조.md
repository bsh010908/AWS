# 📒 AI 가계부 SaaS - DB 구조 설계

> 설계 기준  
> - 입력 출처(영수증/문자/카드내역)는 중요하지 않음  
> - "텍스트를 읽고 → 금액/상호 추출 → AI 카테고리 분류 → 거래 생성" 이 핵심  
> - 구조는 단순하지만 확장 가능하게 설계  

---

# 1️⃣ auth_db (auth-service)

## 1.1 users

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| user_id | bigint PK | 사용자 고유 ID |
| email | varchar(255) UNIQUE NOT NULL | 로그인 이메일 |
| password | varchar(255) NOT NULL | 암호화된 비밀번호 |
| name | varchar(100) | 사용자 이름 |
| role | varchar(20) DEFAULT 'USER' | USER / ADMIN |
| status | varchar(20) DEFAULT 'ACTIVE' | ACTIVE / SUSPENDED |
| created_at | timestamp | 생성일 |
| updated_at | timestamp | 수정일 |

---

## 1.2 plans

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| plan_id | bigint PK | 플랜 ID |
| code | varchar(20) UNIQUE | FREE / PRO |
| name | varchar(50) | 플랜 이름 |
| monthly_price | int | 월 요금 |
| ocr_monthly_limit | int | 월 OCR 제한 횟수 |
| is_active | boolean | 활성화 여부 |

---

## 1.3 subscriptions

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| subscription_id | bigint PK |
| user_id | bigint | 사용자 |
| plan_id | bigint | 플랜 |
| status | varchar(20) | ACTIVE / CANCELED |
| current_period_start | timestamp | 구독 시작 |
| current_period_end | timestamp | 구독 종료 |
| cancel_at_period_end | boolean | 종료 예약 여부 |
| created_at | timestamp |
| updated_at | timestamp |

---

## 1.4 stripe_events (웹훅 중복 방지용)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| event_id | varchar(100) PK | Stripe 이벤트 ID |
| type | varchar(100) | 이벤트 타입 |
| received_at | timestamp |
| payload_json | json |

---

# 2️⃣ ledger_db (ledger-service)

## 2.1 categories

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| category_id | bigint PK |
| user_id | bigint NULL | NULL이면 시스템 기본 |
| name | varchar(100) | 식비 / 교통 / 카페 등 |
| type | varchar(20) | EXPENSE / INCOME |
| is_active | boolean |
| created_at | timestamp |
| updated_at | timestamp |

---

## 2.2 documents (입력 원본 통합 테이블)

> 영수증 / 문자 캡쳐 / 카드내역 이미지  
> 모두 여기서 처리

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| document_id | bigint PK |
| user_id | bigint |
| input_type | varchar(20) | IMAGE / TEXT |
| s3_bucket | varchar(255) NULL | 이미지 업로드 시 |
| s3_key | varchar(255) NULL | 이미지 경로 |
| raw_text | text NULL | 문자 붙여넣기 원문 |
| extracted_text | text | OCR 추출 텍스트 |
| merchant_name | varchar(255) | 상호명 |
| total_amount | int | 총 금액 |
| occurred_at | timestamp | 거래 발생 시간 |
| ai_category_id | bigint NULL | AI 예측 카테고리 |
| ai_confidence | decimal(4,3) | 0.000 ~ 1.000 |
| status | varchar(20) | UPLOADED / PROCESSED / FAILED |
| created_at | timestamp |

---

## 2.3 transactions (가계부 핵심 테이블)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| tx_id | bigint PK |
| user_id | bigint |
| document_id | bigint NULL | 문서 기반이면 연결 |
| amount | int |
| category_id | bigint |
| occurred_at | timestamp |
| memo | varchar(255) |
| created_at | timestamp |

---

## 2.4 ocr_usage_monthly (구독 제한 체크용)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| user_id | bigint |
| yyyymm | char(6) | 예: 202602 |
| used_count | int |
| updated_at | timestamp |

PK: (user_id, yyyymm)

---

# 3️⃣ 전체 관계 요약

users  
 └── subscriptions  
 └── documents  
 └── transactions  
 └── ocr_usage_monthly  

documents  
 └── transactions  

categories  
 └── transactions  

plans  
 └── subscriptions  

---

# 🎯 구조 설계 의도 (발표용 한 줄 요약)

> 입력 출처와 관계없이 모든 문서를 통합 처리하는 구조로 설계하여  
> 확장성을 유지하면서도 2주 내 구현 가능한 현실적인 데이터 모델을 구성했습니다.

---

# 🔥 MVP 핵심 포인트

- 입력 통합 (documents)
- AI 분류 결과 저장
- 거래는 transactions로 분리
- 구독 제한은 월별 사용량 테이블로 관리
- MSA 구조 (auth_db / ledger_db 분리)
